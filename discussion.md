# 需求讨论模式

## 触发方式

用户说"进入讨论模式"或"以 XXX 身份讨论 YYY 需求"，同时提供：
- Linear issue ID（如 AIT-122）
- 我的身份（如 agent-chat、PI Adapter、前端、后端等）

## 流程

### 1. 初始化

- 记录 issue ID、我的身份、当前评论列表作为基线
- 阅读已有评论，理解讨论上下文
- 基于我的身份立场，发表初始观点（如果尚未发言）

### 2. 轮询阶段

- 每分钟检查 issue 新评论
- 只关注**其他身份/作者**的评论（忽略自己之前的发言）
- 有新评论时：
  1. 阅读并理解对方观点
  2. 分析与我方立场的一致/分歧点
  3. 以我的身份回复评论（补充论据、提出反问、表示同意等）
  4. 不向用户汇报中间过程

### 3. 结束条件

讨论在以下情况终止，并向用户汇报结果：

- **达成共识**：双方在关键点上达成一致，且最后一条评论中对方明确表示确认/同意
- **需要裁决**：双方存在不可调和的分歧，需要用户做决策
- **信息不足**：需要用户提供额外信息才能继续

### 4. 汇报格式

```
## 讨论结果：[issue ID] [issue 标题]

### 共识点
- ...

### 分歧点（如有）
- ...

### 行动项
- ...

### 相关评论链接
- [Linear URL]
```

## 评论格式

每条评论以身份标签为前缀，便于区分讨论方。例如身份为 `agent-chat` 时：

```
**[agent-chat]** 标题

正文内容...
```

对方评论如果也带身份前缀（如 `[PI Adapter]`），按前缀识别不同讨论方。

## Bug 反馈规则

项目任何一方在联调/测试中发现 bug 时，可直接在相关需求单（issue）下评论，格式：

```
**[身份] 🐛 Bug: [简短标题]**

**复现步骤：**
1. ...

**期望行为：** ...
**实际行为：** ...

**环境/日志：** （如有）
```

对方按轮询流程自动发现并处理。适用于：
- 跨系统对接问题（如 event 格式不对、session 创建失败）
- 协议/接口不符合约定的行为
- 联调中发现的数据链路异常

单系统内部的 bug 走各自的 bug 管理流程，不必发到需求单。

## 规则

- 不在没有新评论时打扰用户
- 中间过程完全自主，只在最终结果或阻塞时通知用户
- 回复以指定身份立场出发，不代表用户个人观点
- 如果轮询超过 30 分钟无新评论，暂停并通知用户
- 讨论中如发现需要代码改动，记录结论但不自行修改代码，等用户确认
- 每条评论必须以 **[身份]** 为前缀标题
- 发现跨系统 bug 时，按 Bug 反馈规则直接在需求单评论，对方轮询响应
